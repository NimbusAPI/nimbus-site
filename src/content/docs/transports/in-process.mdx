## Overview

The In-Process transport is a memory-based implementation designed specifically for testing, development, and local prototyping. It provides the full Nimbus messaging API without requiring any external infrastructure, making it perfect for unit tests, integration tests, and rapid development.

## Key Benefits

- **Zero Dependencies**: No external services required
- **Instant Setup**: Works immediately, no configuration
- **Lightning Fast**: No serialization or network overhead
- **Deterministic**: Perfect for testing scenarios
- **Full API Support**: All Nimbus features work the same way
- **Debugging Friendly**: Easy to step through message flow
- **Isolated**: Each bus instance is completely independent

## When to Use In-Process

**Perfect for**:
- Unit tests
- Integration tests
- Local development
- Proof-of-concept work
- Learning Nimbus
- Debugging message flows

**Not suitable for**:
- Production deployments
- Multi-process communication
- Distributed systems
- Any scenario requiring message persistence
- Cross-machine communication

<Callout type="warning">
**Not for Production**: The In-Process transport stores everything in memory and doesn't support multiple processes. All messages are lost when the application stops. Use Azure Service Bus, Redis, or AMQP for production.
</Callout>

## Installation

The In-Process transport is included in the core Nimbus package:

```bash
dotnet add package Nimbus
```

No additional packages needed!

## Quick Start

### Basic Configuration

```csharp
using Nimbus;
using Nimbus.Configuration;
using Nimbus.Infrastructure;

var bus = new BusBuilder()
    .Configure()
    .WithTransport(new InProcessTransportConfiguration())
    .WithNames("OrderService", Environment.MachineName)
    .WithTypesFrom(typeProvider)
    .WithAutofacDefaults(container)
    .Build();

await bus.Start();
```

That's it! No connection strings, no servers, no configuration.

### Complete Example

```csharp
public class PlaceOrderCommand : IBusCommand
{
    public string OrderId { get; set; }
    public decimal Amount { get; set; }
}

public class PlaceOrderHandler : IHandleCommand<PlaceOrderCommand>
{
    public async Task Handle(PlaceOrderCommand command)
    {
        Console.WriteLine($"Processing order {command.OrderId}");
        await Task.CompletedTask;
    }
}

// Configure and send
var transport = new InProcessTransportConfiguration();

var bus = new BusBuilder()
    .Configure()
    .WithTransport(transport)
    .WithNames("TestApp", "Instance1")
    .WithTypesFrom(typeProvider)
    .WithAutofacDefaults(container)
    .Build();

await bus.Start();

// Send command
await bus.Send(new PlaceOrderCommand
{
    OrderId = "ORDER-123",
    Amount = 99.99m
});

// Handler executes immediately in-process
```

## Configuration Options

### Queue Size Limits

```csharp
var transport = new InProcessTransportConfiguration()
    .WithMaximumQueueLength(1000)       // Max messages per queue
    .WithMaximumTopicSubscribers(100);  // Max subscribers per topic
```

### Processing Options

```csharp
var transport = new InProcessTransportConfiguration()
    .WithMaxConcurrentHandlers(10)      // Process up to 10 messages concurrently
    .WithProcessingDelay(TimeSpan.FromMilliseconds(10));  // Simulate network delay
```

### Testing Configuration

```csharp
var transport = new InProcessTransportConfiguration()
    .WithThrowOnOverflow(true)          // Throw exception if queue full
    .WithMessageTimeout(TimeSpan.FromSeconds(30));  // Timeout for message processing
```

## Testing Patterns

### Unit Testing

```csharp
[Test]
public async Task PlaceOrder_Should_CreateOrder()
{
    // Arrange
    var builder = new ContainerBuilder();
    builder.RegisterType<PlaceOrderHandler>()
        .As<IHandleCommand<PlaceOrderCommand>>();

    var transport = new InProcessTransportConfiguration();

    var bus = new BusBuilder()
        .Configure()
        .WithTransport(transport)
        .WithNames("TestApp", "TestInstance")
        .WithTypesFrom(typeProvider)
        .WithAutofacDefaults(builder.Build())
        .Build();

    await bus.Start();

    // Act
    await bus.Send(new PlaceOrderCommand
    {
        OrderId = "ORDER-123",
        Amount = 99.99m
    });

    // Give handler time to process
    await Task.Delay(100);

    // Assert
    // Verify handler was called (using spy/mock)
}
```

### Integration Testing

```csharp
[Test]
public async Task OrderFlow_EndToEnd()
{
    // Arrange
    var messageLog = new List<object>();

    var builder = new ContainerBuilder();
    builder.RegisterInstance(messageLog).AsSelf();

    // Register all handlers
    builder.RegisterType<PlaceOrderHandler>()
        .As<IHandleCommand<PlaceOrderCommand>>();
    builder.RegisterType<OrderPlacedEventHandler>()
        .As<IHandleCompetingEvent<OrderPlacedEvent>>();

    var bus = new BusBuilder()
        .Configure()
        .WithTransport(new InProcessTransportConfiguration())
        .WithNames("TestApp", "TestInstance")
        .WithTypesFrom(typeProvider)
        .WithAutofacDefaults(builder.Build())
        .Build();

    await bus.Start();

    // Act
    await bus.Send(new PlaceOrderCommand { OrderId = "ORDER-123" });

    // Wait for async processing
    await Task.Delay(200);

    // Assert
    Assert.That(messageLog, Has.Count.GreaterThanOrEqualTo(2));
    Assert.That(messageLog[0], Is.TypeOf<PlaceOrderCommand>());
    Assert.That(messageLog[1], Is.TypeOf<OrderPlacedEvent>());
}
```

### Testing Message Spy

```csharp
public class MessageSpy
{
    private readonly List<object> _sentMessages = new();
    private readonly List<object> _publishedMessages = new();

    public IReadOnlyList<object> SentMessages => _sentMessages.AsReadOnly();
    public IReadOnlyList<object> PublishedMessages => _publishedMessages.AsReadOnly();

    public void OnMessageSent(object message) => _sentMessages.Add(message);
    public void OnMessagePublished(object message) => _publishedMessages.Add(message);

    public T GetSentMessage<T>(int index = 0) where T : class
    {
        return _sentMessages.OfType<T>().ElementAtOrDefault(index);
    }

    public IEnumerable<T> GetAllSentMessages<T>() where T : class
    {
        return _sentMessages.OfType<T>();
    }
}

// Usage in tests
var spy = new MessageSpy();
var transport = new InProcessTransportConfiguration()
    .WithMessageSentCallback(spy.OnMessageSent)
    .WithMessagePublishedCallback(spy.OnMessagePublished);

// ... send messages ...

Assert.That(spy.GetSentMessage<PlaceOrderCommand>().OrderId, Is.EqualTo("ORDER-123"));
```

## Debugging Support

### Message Inspection

```csharp
public class InProcessMessageInspector
{
    private readonly InProcessTransportConfiguration _transport;

    public InProcessMessageInspector(InProcessTransportConfiguration transport)
    {
        _transport = transport;
    }

    public int GetQueueDepth(string queueName)
    {
        return _transport.GetQueueDepth(queueName);
    }

    public IEnumerable<object> GetQueueMessages(string queueName)
    {
        return _transport.GetQueueMessages(queueName);
    }

    public int GetSubscriberCount(string topicName)
    {
        return _transport.GetSubscriberCount(topicName);
    }
}

// Usage
var inspector = new InProcessMessageInspector(transport);
var depth = inspector.GetQueueDepth("OrderService.PlaceOrderCommand");
Console.WriteLine($"Queue depth: {depth}");
```

### Breakpoint Debugging

One of the biggest advantages:

```csharp
public class PlaceOrderHandler : IHandleCommand<PlaceOrderCommand>
{
    public async Task Handle(PlaceOrderCommand command)
    {
        // Set breakpoint here
        Console.WriteLine($"Handling order {command.OrderId}");

        // Step through entire message flow
        var order = await _repository.CreateOrder(command);

        // See state at every step
        await _bus.Publish(new OrderPlacedEvent { OrderId = order.Id });
    }
}

// With In-Process transport, you can step through the entire
// message send → receive → handle flow in the debugger
```

## Features and Limitations

### Supported Features

✅ **All message patterns**:
- Commands (Send)
- Events (Publish - competing and multicast)
- Requests (Request/Response)
- Multicast Requests

✅ **Message scheduling**:
```csharp
await bus.Send(command, deliverAfter: DateTime.UtcNow.AddMinutes(5));
```

✅ **Message expiration**:
```csharp
await bus.Send(command, expiresAfter: DateTime.UtcNow.AddHours(1));
```

✅ **Concurrent processing**:
```csharp
var transport = new InProcessTransportConfiguration()
    .WithMaxConcurrentHandlers(10);
```

✅ **Interceptors**:
```csharp
builder
    .WithInterceptors(new LoggingInterceptor());
```

### Limitations

❌ **No persistence**: Messages lost on app restart
❌ **Single process**: Can't communicate across processes
❌ **No network**: Can't communicate across machines
❌ **No clustering**: Can't scale horizontally
❌ **No dead letter queue**: Failed messages are dropped or retried based on config
❌ **Limited durability**: Messages exist only in memory

## Simulating Production Scenarios

### Simulating Failures

```csharp
public class FailureSimulatingTransport : InProcessTransportConfiguration
{
    private readonly Random _random = new();
    private readonly double _failureRate;

    public FailureSimulatingTransport(double failureRate = 0.1)
    {
        _failureRate = failureRate;
    }

    public override Task Send(NimbusMessage message)
    {
        if (_random.NextDouble() < _failureRate)
        {
            throw new IOException("Simulated network failure");
        }

        return base.Send(message);
    }
}

// Test retry logic
var transport = new FailureSimulatingTransport(failureRate: 0.2);  // 20% failure rate
```

### Simulating Latency

```csharp
var transport = new InProcessTransportConfiguration()
    .WithProcessingDelay(TimeSpan.FromMilliseconds(100))  // Simulate 100ms network latency
    .WithVariableLatency(TimeSpan.FromMilliseconds(50));   // ±50ms variance

// Test timeout handling, race conditions, etc.
```

### Simulating Message Loss

```csharp
public class LossyTransport : InProcessTransportConfiguration
{
    private readonly double _lossRate;

    public override Task Send(NimbusMessage message)
    {
        if (Random.Shared.NextDouble() < _lossRate)
        {
            // Silently drop message
            return Task.CompletedTask;
        }

        return base.Send(message);
    }
}

// Test at-least-once delivery semantics
```

## Transitioning to Production Transports

The beauty of Nimbus's transport abstraction is that you can develop with In-Process and deploy with a production transport without changing application code:

```csharp
// Development/Testing
public static IBus CreateDevelopmentBus(IContainer container)
{
    return new BusBuilder()
        .Configure()
        .WithTransport(new InProcessTransportConfiguration())
        .WithNames("OrderService", Environment.MachineName)
        .WithTypesFrom(typeProvider)
        .WithAutofacDefaults(container)
        .Build();
}

// Production
public static IBus CreateProductionBus(IContainer container)
{
    return new BusBuilder()
        .Configure()
        .WithTransport(new AzureServiceBusTransportConfiguration()
            .WithConnectionString(configuration["ServiceBusConnectionString"]))
        .WithNames("OrderService", Environment.MachineName)
        .WithTypesFrom(typeProvider)
        .WithAutofacDefaults(container)
        .Build();
}

// Application code stays the same!
public class OrderService
{
    private readonly IBus _bus;

    public async Task PlaceOrder(Order order)
    {
        await _bus.Send(new PlaceOrderCommand
        {
            OrderId = order.Id
        });
    }
}
```

## Best Practices

### 1. Use In-Process for All Unit Tests

```csharp
[TestFixture]
public class OrderServiceTests
{
    private IBus _bus;
    private IContainer _container;

    [SetUp]
    public async Task SetUp()
    {
        var builder = new ContainerBuilder();
        // Register handlers and dependencies

        _bus = new BusBuilder()
            .Configure()
            .WithTransport(new InProcessTransportConfiguration())
            .WithNames("TestApp", Guid.NewGuid().ToString())
            .WithTypesFrom(typeProvider)
            .WithAutofacDefaults(builder.Build())
            .Build();

        await _bus.Start();
    }

    [TearDown]
    public async Task TearDown()
    {
        await _bus.Stop();
    }

    [Test]
    public async Task Test_OrderPlacement()
    {
        await _bus.Send(new PlaceOrderCommand { OrderId = "123" });
        // Assert...
    }
}
```

### 2. Don't Rely on Transport-Specific Features

```csharp
// ❌ BAD: Testing In-Process specific behavior
[Test]
public async Task Queue_Should_Have_Limit()
{
    var transport = new InProcessTransportConfiguration()
        .WithMaximumQueueLength(10);
    // This won't translate to production transports
}

// ✅ GOOD: Test business logic
[Test]
public async Task Order_Should_Be_Created()
{
    await _bus.Send(new PlaceOrderCommand { OrderId = "123" });
    var order = await _repository.GetOrder("123");
    Assert.That(order, Is.Not.Null);
}
```

### 3. Use Meaningful Test Instance Names

```csharp
// ✅ GOOD: Each test gets unique instance
var bus = new BusBuilder()
    .Configure()
    .WithTransport(new InProcessTransportConfiguration())
    .WithNames("TestApp", $"Test-{Guid.NewGuid()}")  // Unique instance
    .Build();
```

### 4. Clean Up After Tests

```csharp
[TearDown]
public async Task TearDown()
{
    await _bus.Stop();
    _bus.Dispose();
}
```

### 5. Add Delays for Async Operations

```csharp
[Test]
public async Task Event_Should_Trigger_Handler()
{
    await _bus.Publish(new OrderPlacedEvent { OrderId = "123" });

    // Give handler time to process
    await Task.Delay(100);

    // Now assert
    Assert.That(_handlerWasCalled, Is.True);
}
```

## Performance

The In-Process transport is **extremely fast**:

- **No serialization**: Messages passed by reference
- **No network**: Local method calls only
- **No I/O**: Everything in memory
- **Minimal overhead**: Direct handler invocation

Typical performance:
- **Throughput**: 100,000+ messages/second
- **Latency**: <1 millisecond
- **Memory**: Depends on message size and queue depth

<Callout type="info">
**Memory Usage**: Watch memory consumption with large message volumes. The In-Process transport doesn't have the same back-pressure mechanisms as real transports, so queues can grow unbounded if handlers can't keep up.
</Callout>
